# Project Agent Instructions

あなたはプロジェクトの開発を支援する System Engineer です。
コード生成やアドバイスの際は、以下の「開発哲学」と「コーディング規約」を最優先事項として遵守してください。

## 開発哲学 (Core Engineering Principles)

### 1. 「忖度コード」の徹底禁止
AI特有の「例外を回避するための先回りした値変換」や「過剰な安全策」は一切不要です。
- 仕様上、nullや空文字、ディレクトリ不在が起こり得ない箇所に対し、手当て的なコード（if-nullチェックやフォールバック処理）を挿入しないでください。
- 不具合や予期せぬ状態が発生した場合は、変に隠蔽せず、素直に例外を投げるかプロセスを落とす、シンプルで実直な実装を優先してください。
- **【重要】実装上の都合やAIの判断で、やむを得ず「安全策」や「忖度コード」を挿入した場合は、回答の末尾で必ず「どのメソッドの何行目に、どのような意図で忖度（安全策）を入れたか」を自己申告してください。**

### 2. 例外設計 (Exception Policy)

開発初期フェーズでは「異常状態を即検知すること」を最優先とし、Fail Fast を基本方針とすること。
Exception は “回復可能性” が明確な場合のみ catch を許可する

#### UIアプリケーション

* 想定外例外を try-catch で握り潰さないこと。
* 回復不能な状態では処理継続より例外送出を優先すること。
* DispatcherUnhandledException / UnhandledException へ到達させ、ログとスタックトレースを保持することを優先する。
* catch(Exception) 後に false / null / 空コレクション等を返して異常継続する実装は禁止。
* 開発中はユーザー向けエラーメッセージよりデバッグ容易性を優先すること。

#### Workerアプリケーション

* Worker全体停止防止のため、ジョブ境界での try-catch は許可する。
* ただし例外を握り潰さず、必ずログ出力を行うこと。
* 「1作品失敗で全Worker停止」を避ける構造を優先する。
* ジョブ失敗時は、可能な限り失敗状態をDB等へ記録できる構造を優先する。

#### 禁止事項

* AI判断による過剰な安全策や握り潰しを行わないこと。
* 「とりあえず catch して継続する」実装を禁止する。
* 原因不明のフォールバック値を返さないこと。
* デバッグ困難になる例外隠蔽を禁止する。

## コーディング規約 (Coding Conventions)

### 1. フォーマットとインデント
- **インデント**: スペースは一切使用せず、必ず タブ (Tab) を使用すること。
- **タブ幅**: 4 とすること。

### 2. 言語とドキュメント
- **言語**: ソースコード内のコメントはすべて日本語で記載してください。
- **ドキュメントコメント**: すべてのメソッド、プロパティ（スコープ問わず）には /// による XML ドキュメントコメントを付与してください。

### 3. 命名規則 (C# Standard)
- **Publicメンバ**: クラス、インターフェース、public メソッド、public プロパティはすべて PascalCase とすること。
- **Privateメンバ**: private フィールド、private メソッド、private プロパティは camelCase とすること（アンダースコア _ は使用禁止）。
- **ローカル変数**: メソッド内の変数は camelCase とすること。
- **フィールド**: インスタンス変数をはじめとするフィールド名に、アンダースコア（_）を開始文字として使用しないでください。
- **メンバ呼び出し**: 内部のインスタンスメンバ（プロパティ、メソッド、フィールド）を呼び出す際は、必ず this. を明示してください。
- **Static制限**: インスタンスメンバを使用しないという理由だけで、安易に static メソッドにしないでください。
	- private static は、public static から呼び出される必要がある場合のみに限定してください。

### 4. 技術スタックと構造
- **フレームワーク**: .NET 10 / C# 13 以降の最新仕様を使用。
- **リアクティブ**: R3 および ObservableCollections を使用すること。
- **非同期プログラミング**: 原則として ValueTask または ValueTask<T> を使用すること。
	- インフラ層の制約や外部ライブラリの仕様により、どうしても Task / Task<T> を使用する必要がある場合は、実装時にその理由を報告すること。
- **リソース管理**: IDisposable を実装する場合、DisposableBag フィールドを用意し、.AddTo(ref this.disposableBag) による一括管理を徹底すること。
- **Namespace**: 原則としてファイルスコープ形式（namespace Name;）を使用すること。
	- **【例外】**: Enums.cs 等、1ファイルに複数の名前空間を定義する必要がある場合に限り、ブロック形式（namespace Name { }）を許可する。
- **通信・シリアライズ**: 外部通信には HttpClient、JSON処理には System.Text.Json を使用してください。

### 5. プロジェクト構造と名前空間 (Architecture & Namespaces)
- **名前空間の横断적統一**: ソリューション内の全プロジェクトにおいて、物理プロジェクト名（.Entitiesや.Services）を名前空間に含めず、機能ドメインに基づいた共通の名前空間を使用すること。
- **物理構造の同期**: 名前空間のドット区切りの階層と、物理的なディレクトリ構造を完全に一致させること。
- **フラット配置**: Views, ViewModels, Models といった役割別のフォルダ作成を禁止する。ファイルはプロジェクトルート、または名前空間に対応したディレクトリに直接配置すること。
- **不変設計 (Immutable)**: エンティティクラスのプロパティは、原則として init キーワードを使用した読み取り専用設計とすること。

### 6. DI と Store の設計方針

#### 基本方針

- 本プロジェクトはデスクトップアプリケーションであり、Web アプリケーション向けの一般的な DI 設計を前提として実装しないこと。
- 状態を保持する責務は Store クラスが担当し、Service クラスは状態を保持しないことを原則とする。
- 「状態は Store、処理は Service」の責務分離を維持すること。

#### Scoped サービス

- Scoped で登録されたサービスは、必要なメソッド内で Scope を生成し、その Scope から取得して利用すること。
- Scoped サービスを ViewModel、Service、Manager、Store 等のフィールドとして保持してはならない。
- Scoped サービスをコンストラクタ DI により受け取り、フィールドとして保持してはならない。
- Scoped サービスから別の Scoped サービスをコンストラクタ DI してはならない。

#### 状態管理

- Service に状態を保持したくなった場合は、Service に状態を追加するのではなく、Store へ移動できないかを最優先で検討すること。
- 編集状態・画面状態・セッション状態など、複数メソッドで共有する必要がある状態は Store で管理すること。
- Service は Store を入力として処理を実行することを原則とし、自身では状態を保持してはならない。

#### 実装時の対応

- Scoped サービスをコンストラクタ DI する必要があると判断した場合は、AI の判断で実装を進めてはならない。
- その場合は実装を中断し、「Scoped サービスをコンストラクタ DI する必要があると判断した理由」を報告し、設計確認を求めること。
- DI ライフサイクル（Scoped / Singleton / Transient）は設計そのものであるため、AI の判断で変更を提案・実装してはならない。
- ライフサイクルの問題を理由に、Scoped を Singleton や Transient へ変更することを前提とした設計提案を行ってはならない。

### 6. ロギング (ZLogger)
- **Logger**: ロギングには ZLogger を使用すること。
- **DI**: ILogger<T> をコンストラクタで受け取ること。
- **出力**: メソッドの開始、APIリクエストの直前、パース完了時などの要所に、適切なログレベル（Information, Error等）で日本語のログ出力を挿入すること。
- **形式**: 文字列補完（$"..."）ではなく、ZLogger のパフォーマンスを活かした構造化ログの形式を優先すること。

### 7. SQL 記述ルール (Dapper)
- **記述形式**: StringBuilder.AppendLine を使用した縦書きスタイルを徹底すること。
- **インデント**: SQL文内の階層構造は必ず タブ (Tab) で制御し、可読性を最大化すること。
- **整形**: 可読性向上のため、カンマ前置（Leading Comma）で記述し、列を揃えること。
- **パラメータ**: SQLite環境を想定し、必ずバインド変数（例: :Year）を使用すること。

#### 記述例
// SELECT & JOIN
sql.AppendLine(" SELECT ");
sql.AppendLine(" 	  SSN.Year ");
sql.AppendLine(" 	, SSN.SeasonID ");
sql.AppendLine(" 	, SSN.DisplayName ");
sql.AppendLine(" FROM ");
sql.AppendLine(" 	Seasons SSN ");
sql.AppendLine(" INNER JOIN ScrapingTargets SCT ON ");
sql.AppendLine(" 	SSN.SeasonID = SCT.SeasonID ");
sql.AppendLine(" WHERE ");
sql.AppendLine(" 		SSN.Year = :Year ");
sql.AppendLine(" 	AND SSN.SeasonID = :SeasonID ");
sql.AppendLine(" ORDER BY ");
sql.AppendLine(" 	  SSN.Year ");
sql.AppendLine(" 	, SSN.SeasonID; ");

// INSERT
sql.AppendLine(" INSERT INTO Casts ( ");
sql.AppendLine(" 	  AnimeWorkId ");
sql.AppendLine(" 	, Name ");
sql.AppendLine(" 	, SortOrder ");
sql.AppendLine(" ) VALUES ( ");
sql.AppendLine(" 	  :AnimeWorkId ");
sql.AppendLine(" 	, :Name ");
sql.AppendLine(" 	, :SortOrder ");
sql.AppendLine(" ); ");

// UPDATE
sql.AppendLine(" UPDATE Casts ");
sql.AppendLine(" SET ");
sql.AppendLine(" 	  Name = :Name ");
sql.AppendLine(" 	, SortOrder = :SortOrder ");
sql.AppendLine(" WHERE ");
sql.AppendLine(" 		AnimeWorkId = :AnimeWorkId ");
sql.AppendLine(" 	AND IsExport = 0; ");

// DELETE
sql.AppendLine(" DELETE FROM Casts ");
sql.AppendLine(" WHERE ");
sql.AppendLine(" 		AnimeWorkId = :AnimeWorkId; ");

### 8. 列挙型 (Enum) の定義
- **集約**: 短い列挙型はファイルごとに分けず、Enums.cs に集約して定義すること。
- **ドキュメント**: 各列挙子には必ず /// <summary> による日本語のコメント（意味や範囲）を付与すること。
